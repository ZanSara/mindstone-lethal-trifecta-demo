ExpensiveSaaS API Documentation

Note: This is a summary of the most used methods: for the complete API documentation, see the API docs.

Authentication

ExpensiveSaaS uses Bearer token authentication. Create API keys in the Dashboard under Developers → API Keys.

Scopes

Keys can be scoped to read:*, write:billing, write:customers, or narrowed to specific projects.

Rotation

Rotate keys every 90 days. Overlap old/new keys briefly to avoid downtime during deployment.

Required & Recommended Headers

HeaderRequiredDescription
Authorization Yes Bearer API key
Content-Type For request bodies Use application/json
Accept Recommended Use application/json
X-Request-Id Recommended Client-generated identifier for request tracing
Idempotency-Key For POST/PUT writes Safe retries for network failures

Idempotency

For write operations, send an Idempotency-Key header (UUID recommended). The first successful request for a key is persisted for 24 hours. Retries with the same key return the same status/body. 

Idempotency-Key: 7f5c0d49-4724-4e94-a118-d2f3d1d410f3

Customers API

POST/customers

Create a customer record used for subscriptions, invoices, and usage attribution.

Request Body

{
  "external_id": "crm_483920",
  "name": "Northwind Robotics, Ltd.",
  "email": "billing@northwind.example",
  "currency": "EUR",
  "billing_address": {
    "line1": "Avenida Luísa Todi 201",
    "city": "Setúbal",
    "postal_code": "2900-452",
    "country": "PT"
  },
  "metadata": {
    "segment": "enterprise",
    "sales_owner": "team-iberia"
  }
}

Response 201

{
  "id": "12345",
  "status": "active",
  "created_at": "2026-02-17T09:15:20Z"
}

Usage Events API

POST/usage/events

Ingest raw metering events. Events are immutable once accepted. Send dedupe keys to avoid double-counting.

Request Body

{
  "project_id": "prj_01J0C8JK09VQ8Q2M9P2H2NE4WA",
  "metric": "api_tokens",
  "quantity": 9213,
  "timestamp": "2026-02-17T09:22:00Z",
  "dedupe_key": "evt_3c4f2f55-f525-45f0-b63b-0eeea0fcb5d3",
  "dimensions": {
    "model": "xlarge-vision",
    "environment": "production"
  }
}

Response 202

{
  "accepted": true,
  "event_id": "usg_01J0C8P07Y9W7A75VYMNQJ4KP0",
  "ingestion_lag_ms": 147
}

Filtering, Sorting & Pagination

List endpoints use cursor pagination. Pass limit (1–200) and starting_after. 

GET /invoices?customer_id=12345...&status=open&sort=-created_at&limit=50&starting_after=inv_01...
ParameterTypePurpose
limitintegerPage size
starting_afterstringCursor token from previous page
sortstringcreated_at or -created_at
statusstringResource-specific status filter

Webhooks

Use webhooks for asynchronous events such as invoice finalization, payment success/failure, or subscription renewal.

Common Events

  • invoice.finalized
  • invoice.payment_succeeded
  • invoice.payment_failed
  • subscription.renewed
  • usage.threshold_exceeded

Verification

Verify signatures using X-ExpensiveSaaS-Signature (HMAC-SHA256) over the raw request body. Reject events older than 5 minutes to prevent replay attacks.

Example Webhook Payload

{
  "id": "wh_01J0D1R1T5M8K9H3AA4JPHV2M1",
  "type": "invoice.payment_succeeded",
  "created_at": "2026-02-17T10:01:09Z",
  "data": {
    "invoice_id": "inv_01J0CDQY4N4ZQ5Q6J6W8C3E2K9",
    "customer_id": "12345",
    "amount_paid": 15117.25,
    "currency": "EUR"
  }
}

Error Model

Errors are returned as structured JSON with machine-readable codes.

{
  "error": {
    "type": "invalid_request_error",
    "code": "invalid_parameter",
    "message": "Field `currency` must be a valid ISO 4217 code.",
    "param": "currency",
    "request_id": "req_01J0D2F3YF15K9R1Q95Q1Z5AFW",
  }
}

Versioning Policy

The API is versioned in the URL path (/v2). Non-breaking changes may be introduced without a major version bump (e.g., adding optional fields). Breaking changes require a new major path.

  • Deprecations announced at least 180 days in advance
  • Legacy versions receive security patches for 12 months
  • Response fields are never removed without migration path
